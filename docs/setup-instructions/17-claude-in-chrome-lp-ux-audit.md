# 17. Claude in Chrome に渡す LP / 登録ファネルの UX 監査指示書

ケンミツの広告ファネル（広告クリック → LP → 登録 → 利用 → 課金）の **LP・登録周りの UX を実機で評価**してもらう指示書。広告で 20 クリック → 1 登録（CVR 5%）と中央値水準だが改善余地あり、という前提で「どこが落ちてるか」を切り分ける目的で使う。

**Claude in Chrome（または Chrome 拡張版 Claude）** にこのファイル全体を貼り付けて「指示通り実行して」と依頼すれば、ブラウザを操作して各ページを開き、評価レポートを返してくれる。

ユーザーに求められる作業:
- 評価結果の確認のみ（実装は別途）
- 万一テスト用メアド登録を依頼された場合のみ承認

---

## 重要な前提（Claude in Chrome に伝えるコンテキスト）

- **コードでの実装は今回しない**。あくまで UX 評価レポートを返してもらうだけ
- LP は別途**大規模刷新予定**なので、細かい LP 内部の修正提案より「**どの導線・どのコピー・どの摩擦が CVR を落としてるか**」という構造的観察を期待
- 改善提案は**優先度（Hi/Mid/Lo）と実装コスト（小/中/大）と想定効果（CVR 改善幅）**を必ず付ける
- ターゲット: 建設業 SMB / 一人親方（ITリテラシー高くない・モバイル使用率高い）
- ピボット後の戦略: 「無料で使える見積書ツール」で集客 → ツール内体験 + ドリップメール教育で時間差課金転換。LP 直接課金訴求は弱め
- 価格: 有料プラン ¥1,980/月（年額 ¥19,800）

## タスク 2: 同じ LP をモバイルで評価（重要）

建設業者は現場でモバイル閲覧が多い。モバイル UX は CVR に直結する。

1. Chrome DevTools を開く（Cmd+Option+I or F12）
2. 「**Toggle device toolbar**」（Cmd+Shift+M）でモバイル表示に切り替え
3. デバイスを **iPhone 14 Pro** または **Pixel 7** に設定
4. ページを再読み込み
5. 以下を観察:
   - **ヒーローの CTA がスクロールせずに見えるか**
   - **テキストが読める大きさか**（建設業者は老眼の方も多い）
   - **タップ可能エリアが指サイズか**（最低 44x44 px）
   - **横スクロールが発生してないか**（レイアウト崩れ）
   - **画像の読み込み速度に違和感ないか**
6. メモ（5 行以内）:
   - モバイル CTA の到達性
   - 文字サイズ・読みやすさ
   - レイアウト崩れの有無
   - PC 版と比べて劣化してる箇所
   - モバイル独自の問題（あれば）

## タスク 7: 統合レポート作成

タスク 1〜6 のメモを統合して以下の形式で**最終レポート**を返してください:

### Part A: 摩擦ポイントの仮説（CVR を落としてそうな順）

各項目に以下を付ける:
- **どの段階か**（広告→LP / LP→/start / /start→メアド送信 / メール認証 / mypage→ツール / ツール→課金）
- **症状**（どう離脱しそうか）
- **原因の仮説**（なぜそうなるか）

例:
```
1. [LP→/start] CTA が画面下 30% スクロールしないと見えない
   → 直帰率が高くなる
   原因: ヒーロー CTA がテキストリンクで弱い
```

### Part B: 改善提案（優先度別）

各提案に必ず付ける:
- **優先度**: Hi / Mid / Lo
- **実装コスト**: 小（30 分以内）/ 中（半日）/ 大（要設計）
- **想定 CVR 改善幅**（例: +20-30%）
- **対応する摩擦ポイント**（Part A の番号と紐付け）
- **実装方針の概略**（具体的にどこを変えるか）

最低 5 個、最大 10 個まで提案。

### Part C: 全体所感

3-5 行で、ファネル全体の所感（強い箇所・弱い箇所・優先順）を述べる。

レポートは Markdown 形式で読みやすく整形して提示してください。
