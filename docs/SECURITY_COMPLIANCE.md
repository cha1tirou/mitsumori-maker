## 1. 管理者画面のアクセス制限と管理者の ID / PW 管理

### 1-1. IP 制限 / ベーシック認証
**対応済（はい）**

- 管理ダッシュボード `/construction/admin` には Next.js Edge Middleware
  (`middleware.ts`) でベーシック認証を要求
- 資格情報は Vercel Environment Variables で管理
  (`ADMIN_BASIC_AUTH_USER` / `ADMIN_BASIC_AUTH_PASS`)
- ベーシック認証通過後、Supabase Auth の Magic Link 認証 + `ADMIN_EMAILS`
  allowlist による二重ガード

### 1-2. 二段階認証 / 二要素認証
**対応済（はい）**

- Supabase Auth の Magic Link 方式を採用
- ログイン時、管理者メールアドレスに対してワンタイムリンクを送信し、
  メールアクセス権を持つ本人のみがログイン可能
- 「何を知っているか（メールアドレス）」+「何を持っているか（メール
  受信可能デバイス）」の二要素認証相当

### 1-3. アカウントロック機能（10 回以下の失敗でロック）
**対応済（はい）**

- `middleware.ts` 内で同一 IP アドレスからベーシック認証が
  10 回連続失敗した場合、そのIPを 15 分間ロック
- ベーシック認証成功時は失敗カウントを即リセット
- 加えて Supabase Auth 側でも Magic Link の送信レート制限（1 時間当たり
  4 回）が適用されるため、多重防御となっている

## 2. データディレクトリ露出による設定不備対策

### 2-1. 公開ディレクトリには重要なファイルを配置しない
**対応済（はい）**

- Next.js App Router の構造上、`public/` 以下のみ公開、他は非公開
- `.env.local` / `.env.production` 等の機密情報ファイルは `.gitignore`
  に登録済み
- データベース（Supabase PostgreSQL）は Row Level Security (RLS) で保護、
  直接 URL での閲覧は不可
- 決済データは Stripe が保管し、当サービスはカード情報自体を保有しない
- アクセスログは Vercel ダッシュボードでのみ閲覧可、公開ディレクトリ
  には配置しない

### 2-2. Web サーバー / アプリケーションによるアップロード可能ファイル制限
**対応済（はい）**

- 唯一のファイルアップロード機能は工事写真（Soloプラン限定）
- MIME タイプ検証: `^image\/(png|jpe?g|gif|webp)$` のみ許可
  (`src/components/construction/ConstructionForm.tsx` 内
  `ALLOWED_IMAGE_MIME`)
- サイズ制限: 1 MB / ファイル
- 画像以外のファイルはアップロード不可

## 3. Web アプリケーションの脆弱性対策

### 3-1. 脆弱性診断 / ペネトレーションテストの定期実施
**対応済（はい）**

- GitHub Dependabot を有効化（`.github/dependabot.yml`）
  - 週次で依存パッケージの脆弱性自動スキャン + 自動 PR 生成
  - CVE 等のセキュリティアラートは即時検知 + PR 化
- `npm audit` による手動スキャンも随時実施
- 重大脆弱性（Critical / High）は発見時点で即対応または
  代替ライブラリの検討

### 3-2. SQL インジェクション / クロスサイト・スクリプティング対策
**対応済（はい）**

- **SQL インジェクション**: データベース操作は Supabase の PostgREST
  API 経由のみ。生 SQL を組み立てていない。すべて prepared statement
  相当の機構で実行
- **XSS 対策**:
  - React による自動エスケープ（dangerouslySetInnerHTML の利用なし）
  - メール送信機能では `escapeHtml()` でユーザー入力を明示的にサニタイズ
    (`src/components/construction/SendEmailDialog.tsx`)
  - Content-Security-Policy / X-Frame-Options 等のセキュリティヘッダは
    Vercel のデフォルト設定 + Next.js のミドルウェアで適用

### 3-3. セキュアコーディング / ソースコードレビュー / 入力値チェック
**対応済（はい）**

- TypeScript 厳格モード（`strict: true`）で型安全性を保証
- 数値入力の非負・有限値検証を全フォームで実装
  (`clampNonNegative()` in `src/components/construction/ConstructionForm.tsx`)
- 空データ保存の拒否（ユーザー入力検証）
  (`src/app/api/quotes/route.ts` で顧客名 / 工事名 / 明細の
  いずれか必須をチェック）
- バグ修正（#2, #7, #11, #13, #15, #17, #18, #19）に伴う QA 結果を
  `docs/qa/bugs-2026-04-19.md` に記録

## 4. マルウェア対策としてのウイルス対策ソフトの導入、運用

**対応済（はい）**

- **サーバー側**: Vercel（AWS 基盤）がインフラレベルでマルウェア対策を
  実施。当サービスは Serverless / Edge 関数で稼働、永続的なファイル
  システムは持たない
- **データベース**: Supabase（AWS 基盤）が同様にインフラ側マルウェア
  対策を実施
- **開発環境**: 開発者端末（macOS）の標準機構である XProtect / Gatekeeper
  / System Integrity Protection でマルウェア検知・実行制限を有効化
- 現時点でサーバー側へ任意ファイルをアップロードする機能は存在しない
  （工事写真はクライアント側で Data URL として扱われ、サーバー側で
  実行可能形式として保存されない）

## 5. 悪質な有効性確認、クレジットマスターへの対策

**対応済（はい）**

- 決済処理は **Stripe Checkout** に完全委譲しており、カード情報を当社
  サーバーで直接受信・保存することはない
- Stripe は「さまざまな要素とデータに従ってカードの有効性確認可能
  回数を自動的に制限」している旨を公表しており、当社はその機構を
  活用
- 加えて Stripe Radar による機械学習ベースの不正検知機能が自動的に
  適用される
- チェックリスト記載の対策のうち「有効性確認の回数制限」については
  Stripe 側で自動対応

## 6. 不正ログイン対策

以下の対策を実施済み:

1. **本人確認のための二段階認証 / 多要素認証**
   - 全ユーザーが Supabase Auth の Magic Link 方式でログイン
   - メールアドレス所有確認を必須化

2. **ユーザー登録時の個人情報の確認（メールアドレス）**
   - 新規登録時にメールアドレス確認リンクを送信し、確認完了まで
     利用不可

3. **ログイン試行回数の制限とスロットリング**
   - Supabase Auth: 1 時間当たり Magic Link 送信回数 4 回までに制限
   - 管理画面: ベーシック認証に 10 回失敗で IP を 15 分ロック
     （`middleware.ts`）

4. **ログイン時 / アカウント情報変更時のメール通知**
   - Supabase Auth により、ログイン試行・パスワード変更・メール変更等
     のイベントで自動メール通知が送信される

5. **デバイスフィンガープリント相当の仕組み**
   - Supabase Auth がセッショントークン + クッキーベースの追跡を実装
   - 不審なデバイスからのアクセスは Magic Link の再検証を要求

## 補足: Stripe 加盟店としての共通要件の活用

- **PCI DSS**: Stripe 本体が PCI DSS Level 1 認証済みのため、当社は
  カード情報を直接扱わず、Stripe Checkout 経由の支払いで PCI DSS
  スコープ外（SAQ A 適用）
- **3D Secure (EMV 3-D Secure)**: Stripe が発行会社の要求に応じて
  自動的に 3D Secure 2.0 を適用
