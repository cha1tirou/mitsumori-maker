## #21【Major】瑕疵担保責任セクションのページ割れ

### 症状
- 1ページ目末尾: `◆ 瑕疵担保責任・アフターサービス` の見出しのみ残る
- 2ページ目冒頭: 保証期間/構造耐力/内装仕上げ等の本文が始まる

見出しが孤立行になっており可読性・見栄えが悪い。

### 修正方針（実装に応じて選択）

**Puppeteer / HTML→PDF の場合**
- 見出しと直下段落を囲むラッパに `break-inside: avoid;`（レガシーブラウザ互換で `page-break-inside: avoid;` も併記）
- 見出し単独には `break-after: avoid;` を付与すれば直後の段落と切り離されない

**@react-pdf/renderer の場合**
- セクション全体を `<View wrap={false}>` でラップ
- ただし `wrap={false}` は内容が1ページに収まらないと途切れるので、**見出し+最初の段落ペアのみ** `wrap={false}` にする方が安全
- 代替: `<View break>` で明示的な改ページ制御

**weasyprint の場合**
- 同じく `break-inside: avoid` / `break-after: avoid`

### 検証
検証用データで再出力し、見出しと本文が同一ページに存在することを確認。

## #22【Major】SAMPLE透かしのカバレッジ不足

### 症状
2ページ目は下半分（備考の【特記】以降）に大きな余白があるが、その余白にはSAMPLE透かしが入っていない。

### セキュリティ的懸念
透かしなしの領域をトリミングすれば、透かしなしPDFとして流用可能。本番有料版と開発版の差別化を透かしに依存するなら致命的。

### 修正方針
透かしの実装を「1ページ目のみ背景画像」から「全ページの固定レイヤー」に変更する。

- **@react-pdf/renderer**: `<Page>` 直下に `<Image fixed>` または `<Text fixed>` を配置（fixed は全ページに繰り返し描画される）
- **Puppeteer**: `@page` の背景画像にSVG埋め込み、もしくはページ全体を覆うabsolute positionedの透かし要素を各セクションに挿入。`headerTemplate/footerTemplate` は狭く制限されるため透かしには不向き
- **weasyprint**: `@page { background-image: url(...) }` または `@page { @top-left-corner` など margin box を使わず body 内の fixed 要素で対応

推奨パターン: ページ中央に大きなSAMPLE×1 を半透明で配置し、それを全ページ繰り返す。タイル状も可だが視認性を落とさないこと。

### 検証
- 1ページ / 2ページ / 3ページ以上の見積書すべてで、全ページの上下左右すべての領域に透かしが確認できること
- PDFビューアでページを切り取って保存しても必ず透かしが残ること

## #23【Minor】2ページ目のヘッダー／ページ番号不在

### 症状
2ページ目には見積番号・会社名・ページ番号のいずれも表示されていない。物理印刷でページがばらけると出所不明になる。

### 期待動作
2ページ目以降のフッター（もしくはヘッダー）に最低限:
- 見積番号（例: `K-2026-0419`）
- ページ番号（例: `2 / 2`）

1ページ目は既に情報が十分なのでフッタのみでOK（全ページ共通フッタでも可）。

### 修正方針
- **Puppeteer**: `footerTemplate` で以下のような HTML を指定
  ```html
  <div style="font-size:9px; width:100%; text-align:right; padding-right:15mm; color:#666;">
    見積番号 K-2026-0419 &nbsp;&nbsp; <span class="pageNumber"></span> / <span class="totalPages"></span>
  </div>
  ```
  ※ `displayHeaderFooter: true`、`margin.bottom` を 20mm 程度に確保
- **@react-pdf/renderer**:
  ```tsx
  <Text fixed style={styles.footer} render={({ pageNumber, totalPages }) =>
    `見積番号 ${quoteNo}   ${pageNumber} / ${totalPages}`
  } />
  ```
- **weasyprint**: `@page { @bottom-right { content: "見積番号 " var(--quote-no) "  " counter(page) " / " counter(pages); } }`

### 検証
2ページ目フッターに `見積番号 K-2026-0419` と `2 / 2` が表示されること。

## #24【Minor】異体字の不整合検知と警告表示

### 症状
現行テストデータ:
- 発注者名: `髙﨑 誠一`（髙=ハシゴ高、﨑=立崎）
- 工事場所: `東京都渋谷区恵比寿南1-22-8 高﨑ハイツ301号室`（高=標準字体）

発注者姓と建物名で「髙／高」が食い違っているが、入力時に気付けない。typoなのか意図的な別字なのか不明。

### 対応方針（soft warning）

**blockingしない。**フィールド下にグレー文字で警告のみ表示。

発注者名に異体字が含まれている場合、他の全テキストフィールド（工事場所、工事名、備考等）を走査し、同じ字の標準字体が出現したら以下のようなメッセージを表示:

> 発注者名では「髙」を使用していますが、「高﨑ハイツ」は標準字体になっています。意図通りですか？

### 検知対象の異体字マッピング（抜粋）

`lib/ituji-map.ts` などに定義して一元管理する想定:

```ts
export const ITUJI_MAP: Record<string, string[]> = {
  '髙': ['高'],
  '﨑': ['崎'],
  '邊': ['辺', '邉'],
  '邉': ['辺', '邊'],
  '德': ['徳'],
  '澤': ['沢'],
  '齋': ['斎', '斉'],
  '齊': ['斎', '斉'],
  '濵': ['浜', '濱'],
  '濱': ['浜', '濵'],
  '廣': ['広'],
  '寬': ['寛'],
  '惠': ['恵'],
  '眞': ['真'],
  '槇': ['槙'],
  '祐': ['佑'],  // 厳密には異体字ではないが人名で混用されるので候補
};
```

`String.prototype.normalize('NFKC')` では統合されないため、明示的マッピングが必要。

### 実装アウトライン
1. 発注者名入力欄の onBlur で `ITUJI_MAP` のキーに該当する字を抽出
2. 該当字があれば、他フィールドに対してその標準字体が含まれていないか検査
3. 含まれていれば該当フィールドに警告メッセージを出す
4. 警告は閉じるボタンで dismissible

### 実装が重い場合
見送って `bugs-2026-04-19.md` に `WONTFIX (実装コスト要検討)` として記録して良い。Majorではないので優先度低。

### 検証
- `髙﨑 誠一` / `高﨑ハイツ` → 警告表示
- `髙﨑 誠一` / `髙﨑ハイツ` → 警告なし
- `高橋 太郎` / `高橋ハイツ` → 警告なし（そもそも異体字なし）
